** Purpose
This document is the canonical implementation blueprint for an MVP framework that brings
LM program authoring, compilation, optimization, and runtime execution into the .NET
ecosystem for enterprise use, inspired by DSPy's core insight that LM programs should
be optimized programmatically rather than manually tuned.

** Provenance Note
This framework adopts DSPy's foundational concepts (Signatures, Predict, Retrieve,
programmatic optimization) but extends well beyond them with .NET-native innovations.
Major original contributions include: directed program graphs (DSPy uses flat Python
modules), compile-time validation via Roslyn analyzers and source generators,
constraint-based optimization (DSPy has no constraint model), three-tier data flow

** Relationship to DSPy
This framework is /inspired by/ DSPy (Stanford NLP), not a port of it.

From DSPy, we adopt:
- the core insight that LM programs should be optimized programmatically,
- the concept of typed Signatures as LM contracts,
- the ~Predict~ and ~Retrieve~ step abstractions,
- the idea of a compile loop that searches over program variants.

Original to this framework (no DSPy equivalent):
- directed program graphs (DSPy uses flat Python modules with ~forward()~),
- compile-time validation via Roslyn analyzers and source generators,
- expression-tree-based data flow inspection between steps,
- constraint-based optimization with hard/soft constraint enforcement,
- ~Evaluate~, ~If~, and ~Repair~ as first-class program steps,
- model choice and temperature as tunable parameters,
- typed artifact serialization with AOT safety and hot-swap deployment,
- enterprise observability via OpenTelemetry with in-process trace feedback,
- and the entire build-time architecture (analyzers, code fixes, diagnostics).

DSPy's actual architecture (for accurate reference):
- Programs are Python classes inheriting ~dspy.Module~ with a ~forward()~ method.
- Optimizers (formerly "Teleprompters") include ~BootstrapFewShot~, ~COPRO~,
  ~MIPROv2~, ~SIMBA~, ~GEPA~, ~KNNFewShot~, ~LabeledFewShot~.
- Persistence is via ~Module.save()~/~.load()~ — no versioned artifacts.
- There is no constraint model, no graph abstraction, no compile-time validation.

* 1A. The Business Case for Compiled LM Programs

** The Problem: Enterprise AI Is Bleeding Money

Every enterprise deploying LLM-powered applications faces four compounding cost drivers
that manual prompt engineering cannot solve:

*** Pain 1: Prompt Drift — Silent Failure at Enterprise Scale
When LLM behavior changes (model updates, prompt degradation, data distribution shift),
production applications silently fail. Unlike traditional software bugs, LLMs continue
to produce plausible outputs with degraded accuracy — failures go undetected until
significant business impact arises [1]. A ticket-triage system that starts misrouting
tickets triggers SLA violations, escalation labor, and customer churn.
Enterprises report 3–5× more debugging time per incident and 20–40% of AI budgets
wasted on unnecessary model swaps when root causes aren't systematically identified [2].
Manual prompt engineering provides no reproducibility guarantee — each "fix" is ad hoc,
untested against the original optimization criteria, and invisible to auditing.

*** Pain 2: Manual Optimization — Unquantified but Substantial Labor Cost
/LMP team estimate:/ A senior data scientist optimizing one LM program spends ~2 weeks
per optimization cycle across 10+ cycles annually: prompt variants, few-shot examples,
model selection, temperature tuning, evaluation runs. At blended rates ($150/hr ×
80 hrs × 10 cycles + infrastructure), this yields ~$154K/year /per program/.
This estimate is conservative — it excludes opportunity cost, review overhead, and
the labor of rebuilding optimization from scratch after model updates. The key insight
is not the specific dollar amount but that /this labor produces no reproducible artifact/
and must be repeated whenever the underlying model changes.

*** Pain 3: Governance Failure — EU AI Act Fines Up to 7% of Global Turnover
The EU AI Act Article 99 (effective August 2025) establishes tiered penalties [3]:
- Prohibited AI practices: up to €35M or 7% of global annual turnover (whichever is higher)
- High-risk AI obligations: up to €15M or 3% of global annual turnover
- Supplying incorrect information: up to €7.5M or 1% of global annual turnover
AI system providers must maintain technical documentation, traceability of
optimization decisions, and risk management systems with documented testing.
Manual prompt engineering produces NO audit trail, NO versioned configuration,
NO reproducible optimization history. Every manually tuned prompt is a compliance liability.

*** Pain 4: Enterprise AI Failure Rate — 70–85%
Enterprise AI investments show a consistent pattern of operational failure:
- 70–85% of GenAI deployment efforts fail to meet ROI expectations [4]
- Only ~25% of AI initiatives deliver expected ROI [5]
- 88–95% of AI pilots never reach production deployment [6][7]
- 42% of organizations abandoned most AI initiatives in 2025 (up from 17% in 2024) [8]
- GenAI spending projected to grow from $16B (2023) to $143B+ by 2027 (73% CAGR) [9]
- Most AI projects stall at "pilot purgatory" — technically functional but
  operationally immature, unable to demonstrate sustained business value [7]
The root cause is not technology inadequacy but /operational immaturity/: there is
no engineering discipline for managing LM programs as software — no build/test/deploy
lifecycle, no performance regression detection, no cost optimization automation [4].

* 2A. Deep .NET Platform Primitive Map

** Purpose
This section maps specific .NET runtime, language, and SDK primitives to framework
capabilities, organized by lifecycle phase. Every primitive listed here has been
validated as GA (production-ready) unless marked otherwise.

This section exists because the framework's competitive advantage comes not from any
single .NET feature, but from how 50+ primitives /converge/ across the full lifecycle.
These convergences create capabilities that Python fundamentally cannot replicate.

** Phase 1: Build Time — Author, Validate, Generate

*** IIncrementalGenerator (Roslyn)
Discovers ~[LmpSignature]~ and ~[LmpProgram]~ types at compile time.
Generates descriptor classes as ~.g.cs~ files.
Python equivalent: None — runtime metaclass magic only.

*** Roslyn Analyzers + CodeFixProvider
LMP001-006 diagnostics surface errors in the IDE while typing.
Lightbulb fixes for missing descriptions, invalid step names.
Python equivalent: mypy/pylint (external, weaker, no semantic awareness).

*** Three-Tier Data Flow Binding

**** Tier 1: Convention-Based Auto-Binding (60% of cases)
Source generator detects matching property names and types between step outputs
and downstream step inputs. Auto-generates binding code with zero developer ceremony.
Example: ~RetrieveOutput.Documents~ auto-binds to ~ClassifyInput.Documents~.

**** Tier 2: Attribute-Based Explicit Binding (30% of cases)
~[BindFrom("step-name", nameof(Output.Field))]~ on input record properties.
Source generator reads attributes at compile time — zero runtime overhead.
Trivially serializable to artifact JSON.
Precedent: ~[FromBody]~, ~[Column]~, ~[JsonPropertyName]~, ~[BindProperty]~.

**** Tier 3: Interceptor-Based Lambda Binding (10% of cases — C# 14)
Developer writes natural C# lambdas for computed bindings:
~.Bind(ctx => new ClassifyInput { Summary = $"{ctx.OutputOf(step1).Title}: ..." })~
C# 14 interceptor reads the lambda's SYNTAX TREE at compile time (not expression tree),
generates optimized binding code + serializable binding metadata.
No expression tree limitations (can handle any lambda structure).

**** Tier 4: Expression Trees — Runtime Fallback Only
~Expression<Func<T>>~ for dynamically constructed graphs (e.g., deserialized from artifacts).
Same pattern Entity Framework uses for LINQ→SQL translation.
Limitation: single-expression lambdas only (no async, no statements).
Python equivalent: None — lambdas are opaque bytecode.

*** file class (C# 11)
Generated descriptor types stay out of public API surface.
Python equivalent: naming convention only.

*** static abstract members in interfaces (C# 11)
~ISignature.CreateDescriptor()~ — source generators implement factory methods.
Python equivalent: static methods exist but interface enforcement does not.

*** Partial Properties (C# 13)
Source generators can now implement PROPERTIES, not just methods:
~public static partial SignatureDescriptor Descriptor { get; }~
Source generator generates the implementation. Reads more naturally than ~CreateDescriptor()~.
Property access is cached (static readonly semantics).
Python equivalent: None — no compile-time property generation.

*** params ReadOnlySpan<T> (C# 13)
~params ReadOnlySpan<IStep> steps~ — stack-allocated parameter arrays.
~program.AddSteps(step1, step2, step3)~ — zero heap allocation, 30–50% faster
in microbenchmarks (zero bytes allocated vs ~48 bytes per call with ~params T[]~).
Source: Microsoft .NET Blog, "Calling methods is easier and faster with C# 13
params collections," https://devblogs.microsoft.com/dotnet/csharp13-calling-methods-is-easier-and-faster/
Used for all variadic APIs: ~AddSteps()~, ~AddConstraints()~, ~AddEvaluators()~.
Python equivalent: ~*args~ (heap-allocated tuple).

*** Raw string literals (C# 11)
~"""..."""~ for multi-line prompt templates with zero escaping.
~$$"""..."""~ enables interpolation in raw strings.
Python equivalent: triple-quoted f-strings (similar, but no interpolation control).

*** CallerArgumentExpression (C# 10)
Auto-captures constraint expression text for diagnostics at zero runtime cost.
Example: ~rules.Require(m => m.PolicyPassRate == 1.0)~ automatically generates
diagnostic text containing the expression.
Python equivalent: ~inspect~ module (fragile, runtime-only).

*** required keyword (C# 11) + Nullable Reference Types
Compile error if signature field not set. ~string?~ vs ~string~ for optional/required.
Python equivalent: Runtime TypeError / ~Optional[str]~ (advisory only).

*** C# 14 Features (.NET 10 LTS)

**** Extension Members (C# 14)
Extension properties, static methods, and operators — not just extension methods.
Enables richer fluent APIs for graph construction without polluting framework types.
Python equivalent: monkey-patching (runtime, no type safety).

**** Field-Backed Properties (C# 14)
~field~ keyword in property accessors provides direct access to auto-generated backing field.
Enables validation/transformation in property setters without manual backing fields.
Useful for signature field validation: ~set { field = value ?? throw new ArgumentNullException(); }~
Python equivalent: ~@property~ with explicit ~_field~ (more boilerplate).

**** Interceptors (C# 14 — STABLE)
Source generators can intercept specific method calls at compile time and replace them
with generated code. Used by ASP.NET Core Minimal APIs and OpenAPI.
LMP uses interceptors for Tier 3 binding: intercept ~.Bind()~ calls, analyze the lambda's
syntax tree at compile time, generate optimized binding code + serializable metadata.
Python equivalent: None — no compile-time call interception exists.

**** Implicit Span Conversions (C# 14)
~ReadOnlySpan<char>~ parameters accept ~string~ without explicit conversion.
Enables zero-allocation prompt assembly APIs without API surface complexity.
Python equivalent: None (no zero-copy string views in Python).

** Phase 2: Compile Time — Optimize, Select, Emit

*** Records + with expressions
Safe immutable variant generation: ~v2 = v1 with { Temperature = 0.3 }~.
Compiler guarantees all fields are copied. Structural equality for comparison.
Python equivalent: ~dataclasses.replace()~ (no compiler guarantee against field omission).

*** FrozenDictionary / FrozenSet (.NET 8)
Immutable, cache-line-optimized lookup collections for descriptor registries.
~3x faster than Dictionary for read-heavy workloads.
Python equivalent: ~types.MappingProxyType~ (read-only wrapper, not optimized).

*** ImmutableArray<T>
Structurally immutable — unlike ~IReadOnlyList<T>~, wrapping a mutable list is impossible.
Python equivalent: ~tuple~ (limited composability).

*** INumber<T> / Generic Math (C# 11)
~WeightedAverage<T>() where T : INumber<T>~ — generic metric aggregation.
Python equivalent: numpy (external dependency).

*** System.Threading.RateLimiting (.NET 7+)
~TokenBucketRateLimiter~ for API-safe optimization: prevents overwhelming LM APIs
during multi-trial compile loops.
Python equivalent: ~tenacity~ (external).

*** Parallel.ForEachAsync (.NET 6+)
Bounded-concurrency parallel trial execution with ~MaxDegreeOfParallelism~.
Replaces ~Task.WhenAll(candidates.Select(...))~ which has zero concurrency control —
50 candidates = 50 simultaneous API calls = rate limit errors.
Cooperates with resilience pipeline for API safety.
Python equivalent: ~asyncio.Semaphore~ (manual, GIL-limited for CPU work).

*** AlternateLookup (.NET 9+)
~FrozenDictionary<string, T>.GetAlternateLookup<ReadOnlySpan<char>>()~ enables
zero-allocation dictionary lookups with Span keys.
Critical for hot-path step/parameter resolution when parsing from JSON buffers.
Source: https://learn.microsoft.com/dotnet/api/system.collections.frozen.frozendictionary-2.alternatelookup-1
Python equivalent: none (all lookups allocate).

*** PeriodicTimer + TimeProvider (.NET 6+, .NET 8+)
Drift-free, testable periodic scheduling for long-running optimization services.
Example: recompile every 24h against latest evaluation data.
~FakeTimeProvider~ in tests: advance time without sleeping.
Source: https://learn.microsoft.com/dotnet/api/system.threading.periodictimer
Python equivalent: ~threading.Timer~ (manual drift management).

*** HybridCache (.NET 9, enhanced .NET 10)
Combined L1 (in-memory) + L2 (distributed) cache for LM response deduplication.
Same prompt → cached response across optimization trials.
.NET 10 adds tag-based eviction: tag entries with ~trial-{id}~ and ~step-{name}~,
then ~RemoveByTagAsync("trial-42")~ to evict when prompt template changes.
Python equivalent: ~functools.lru_cache~ + Redis (manual wiring, no tag eviction).

*** TimeProvider (.NET 8)
~FakeTimeProvider~ for deterministic trace timestamps in tests.
Python equivalent: ~unittest.mock.patch~ on datetime.

*** BackgroundService
Long-running compile jobs with graceful Ctrl+C cancellation via ~CancellationToken~.
Python equivalent: manual signal handling.

*** Pattern matching + sealed hierarchies
Exhaustive step dispatch: adding a new step kind causes compile errors everywhere.
Python equivalent: isinstance chains (no exhaustiveness checking).

*** SearchValues<T> (.NET 8)
SIMD-optimized token boundary detection for prompt template parsing (~8x faster).
Python equivalent: ~set~ (not SIMD-optimized).

*** System.Numerics.Tensors.TensorPrimitives (.NET 9+)
SIMD-accelerated math operations — zero external dependency:
- ~TensorPrimitives.CosineSimilarity(span1, span2)~ — embedding similarity for Retrieve steps
- ~TensorPrimitives.Sum()~, ~Average()~, ~Max()~ — metric aggregation in compiler
- Hardware-accelerated (AVX 10.2 in .NET 10)
- Works on ~ReadOnlySpan<float>~ — zero allocation
Replaces custom ~INumber<T>~ generic math for most aggregation.
Replaces TorchSharp for embedding similarity (no external dependency).
Python equivalent: numpy (external dependency).

** Phase 3: Runtime — Execute, Trace, Observe

*** System.Threading.Tasks.Dataflow (TPL Dataflow)
Battle-tested graph execution engine built into .NET:
- ~TransformBlock<TInput, TOutput>~ — maps directly to Predict steps
- ~ActionBlock<T>~ — for side-effect steps (evaluation, logging)
- ~BroadcastBlock<T>~ — for fanout (one output → multiple downstream)
- ~JoinBlock<T1, T2>~ — for merging parallel branches
- Built-in backpressure, bounded capacity, cancellation, error propagation
- Links between blocks with predicates (conditional routing = If step)
The LMP runtime COMPILES the IR down to TPL Dataflow blocks rather than
inventing a custom graph executor. This gives parallel execution,
backpressure, and cancellation for free.
Python equivalent: None — manual asyncio orchestration.

*** IChatClient Middleware Pipeline (Microsoft.Extensions.AI)
M.E.AI provides built-in middleware via ~ChatClientBuilder~:
- ~UseOpenTelemetry()~ — automatic OTEL spans for every LM call
- ~UseDistributedCache()~ — response caching via ~IDistributedCache~
- ~UseLogging()~ — structured prompt/completion logging
- ~UseFunctionInvocation()~ — automatic tool calling dispatch
LMP provides ONLY 2 thin middleware (everything else is M.E.AI's job):
- ~UseLmpStepContext()~ — enriches ~Activity.Current~ with LMP tags
  (step name, trial ID, program name) so built-in OTEL traces carry context
- ~UseLmpCostTracking()~ — reads ~ChatResponse.Usage~, accumulates
  token cost per trial for budget constraint evaluation
Python equivalent: None — no middleware pipeline for LLM calls.

*** Keyed DI Services (.NET 8)
~[FromKeyedServices("triage")] IChatClient client~ — multiple named model configs.
One model per step: "gpt-4" for triage, "gpt-4-mini" for evaluation.
Python equivalent: manual registry dictionary.

*** True Parallelism (no GIL)
~Task.WhenAll(retrieve1, retrieve2)~ runs BOTH LM calls concurrently on separate threads.
Python's asyncio is limited by the GIL for CPU-bound scoring between calls.

*** IAsyncEnumerable<T>
Token streaming as first-class iteration: ~await foreach (var chunk in response)~.
Python equivalent: library-specific streaming.

*** System.Threading.Channels
Lock-free, FIFO producer/consumer for pipeline step execution.
Python equivalent: ~asyncio.Queue~ (limited).

*** Span<T> / stackalloc
Zero-allocation prompt assembly in tight optimization loops.
Python equivalent: string concatenation (GC pressure).

*** System.Threading.Lock (.NET 9+)
Dedicated lock type — compiler generates optimized code (no monitor overhead).
~private readonly Lock _lock = new(); lock (_lock) { ... }~
Used for thread-safe descriptor registries, step caches, and metric accumulators.
Python equivalent: ~threading.Lock~ (similar semantics, no compiler optimization).

*** Microsoft.Extensions.Resilience (Polly-based)
Composable resilience pipelines for LM API calls during compile loop:
- Retry with exponential backoff + jitter (transient API failures)
- Circuit breaker (prevent flooding overwhelmed APIs)
- Timeout (cap per-call latency)
Replaces raw ~TokenBucketRateLimiter~ for a more complete resilience story.
Integrates with ~IChatClient~ middleware pipeline.
Python equivalent: ~tenacity~ (retry only, no circuit breaker).

*** ActivitySource + Activity (System.Diagnostics)
Per-step distributed tracing with W3C Trace Context.
Tags for model, tokens, cost. Automatic correlation IDs across steps.
Python equivalent: manual dicts in DSPy.

*** Meter + Histogram / Counter
Custom metrics: ~lm.predict.latency_ms~, ~lm.cost.total_usd~, ~lm.tokens.input~.
Export to Prometheus, Grafana, Azure Monitor.
Python equivalent: manual instrumentation.

*** InMemoryExporter
In-process trace access — the compiler can READ runtime traces programmatically
to feed back into optimization constraint evaluation. Closed feedback loop.
Python equivalent: No standard equivalent.

*** DataAnnotations + IParsable<T> + TypeConverter
Parse chain for LM text→CLR type conversion:
1. ~IParsable<T>.TryParse()~ (.NET 7+) — modern static abstract interface members.
   Supports all built-in numeric types, ~bool~, ~DateTime~, etc.
   Source: https://learn.microsoft.com/dotnet/standard/generics/math
2. ~TypeConverter~ fallback — handles legacy types and custom converters.
~[Required]~, ~[Range]~ for declarative LM output validation.
Output types may implement ~IValidatableObject~ for structured self-validation.
Python equivalent: Pydantic validators (functions) / manual parsing.

*** [GeneratedRegex] (.NET 7+)
AOT-safe, pre-compiled regular expressions via source generation.
Used for JSON extraction from LM responses (stripping markdown code fences).
Zero runtime compilation overhead; SIMD-accelerated on supported hardware.
Source: https://learn.microsoft.com/dotnet/standard/base-types/regular-expression-source-generators
Python equivalent: ~re.compile()~ (interpreted, not pre-compiled).

*** ConcurrentDictionary
Lock-free, thread-safe dictionary for execution context state.
Replaces ~Dictionary<string, object>~ + manual ~lock~ for step outputs,
scores, and branch results during parallel graph execution.
Python equivalent: ~threading.Lock~ + ~dict~ (manual, GIL-dependent).

*** Covariant/Contravariant generics
~IStep<out TOutput>~ — compile-time verification that step A's output type
matches step B's input type.
Python equivalent: isinstance checks at runtime.

** Phase 4: Deployment — Save, Load, Hot-Swap

*** System.Text.Json + JsonSerializerContext
AOT-safe artifact serialization with source-generated serializers. Zero reflection.
Python equivalent: ~json.dumps~ (no AOT story) / ~pickle~ (security risk).

*** [JsonDerivedType] discriminators
Polymorphic step serialization: ~{"$type": "predict", ...}~.
Handles PredictStep/RetrieveStep/EvaluateStep in artifacts safely.
Python equivalent: pickle (arbitrary code execution vulnerability).

*** AssemblyLoadContext (collectible)
Hot-swap compiled artifacts WITHOUT restarting the service.
Load v2 artifact, unload v1, memory reclaimed.
Python equivalent: ~importlib.reload()~ (leaves old references, rarely used in production).

*** IOptionsMonitor + IChangeToken
Watch artifact files on disk, auto-reload when changed.
Python equivalent: ~watchdog~ (external).

*** System.IO.Hashing
~XxHash128~ for fast artifact provenance hashing.
~SHA256~ for verification.
Python equivalent: ~hashlib~.

*** Native AOT
Compiled programs as native executables: ~50ms cold start vs 2-5s for Python.
Source generators eliminate reflection for AOT compatibility.
Python equivalent: No equivalent.

*** ConfigurationBinder Source Generator (.NET 8+)
~<EnableConfigurationBindingGenerator>true</EnableConfigurationBindingGenerator>~

* 5. User-Facing Mental Model

** Signature
A /Signature/ defines a typed LM task contract.
/Provenance: Adopted from DSPy (~dspy.Signature~). DSPy uses Pydantic-based annotated/
/schemas. This framework extends the concept with compile-time validation via source/
/generators and analyzers — a capability DSPy cannot offer./

A signature contains:
- instructions,
- typed input fields,
- typed output fields,
- field metadata.

A signature is the smallest authorable unit that can map to an LM request / response contract.

** Program
A /Program/ is a directed graph of steps that together solve a business task.
/Provenance: ORIGINAL. DSPy programs are Python classes with a ~forward()~ method —/
/there is no graph abstraction. The directed graph model is a .NET-native invention/
/that enables compile-time cycle detection, topological execution, and expression-tree/
/data flow analysis./

The canonical program for MVP is a ticket triage program.

** Step
A /Step/ is one node in a program graph.

MVP step kinds:
- ~Predict~ /— Adopted from DSPy (~dspy.Predict~)/
- ~Retrieve~ /— Adopted from DSPy (~dspy.Retrieve~)/
- ~Evaluate~ /— ORIGINAL. ~dspy.Evaluate~ is an external utility, not a program step/
- ~If~ /— ORIGINAL. Does not exist in DSPy/
- ~Repair~ (minimal form) /— ORIGINAL. Loosely inspired by ~dspy.Retry~ but substantially different/

** Graph
A /Graph/ defines control flow and data flow between steps.
/Provenance: ORIGINAL. DSPy has no graph abstraction./

The graph is the authored logical structure of the program.
The graph is later lowered into an internal executable representation.

** Evaluator
An /Evaluator/ scores model outputs or overall program behavior.
/Provenance: Adopted from DSPy (~dspy.Evaluate~). This framework integrates with/
/~Microsoft.Extensions.AI.Evaluation~ which provides production-grade evaluators/
/(groundedness, coherence, fluency, relevance) not available in DSPy./

MVP uses existing evaluation libraries where possible and a framework wrapper where needed.

** Compile Spec
A /Compile Spec/ defines:
- training set,
- validation set,
- tunable parameters,
- metrics,
- constraints,
- optimizer choice.
/Provenance: Partially adopted. DSPy's ~BootstrapFewShot~, ~MIPROv2~, etc. accept/
/training data and metrics. The constraint model (hard/soft constraints with/
/~CallerArgumentExpression~ diagnostics) is ENTIRELY ORIGINAL — DSPy has no/
/constraint system./

** Compiler
The /Compiler/ runs candidate trials over a program and chooses the best valid variant.
/Provenance: Adopted concept. DSPy calls these "Teleprompters" (recently "Optimizers")./
/The ~.compile()~ method returns an optimized Module. This framework extends the concept/
/with constraint enforcement, rate-limited trial execution, artifact provenance, and/
/in-process observability feedback loops./

** Compiled Artifact
A /Compiled Artifact/ is a versioned, serializable representation of:
- the base program,
- selected variant parameters,
- metrics,
- provenance,
- and deployment metadata.
/Provenance: Extended from DSPy. DSPy has ~Module.save()~/~.load()~ for basic/
/persistence. This framework's artifact model (AOT-safe JSON serialization,/
/polymorphic step discriminators, hot-swap via ~AssemblyLoadContext~, provenance/
/hashing) is substantially ORIGINAL./

** Runtime Trace
A /Runtime Trace/ is a structured record of:
- executed steps,
- timings,
- model choices,
- scores,
- diagnostics,
- and outputs.

* 5A. Authoring-Time Architecture: Roslyn, Analyzers, and Source Generators

** Purpose
This section defines the build-time authoring architecture of the framework.

This architecture is one of the main reasons this framework belongs in .NET.
It is not optional polish.
It is a core part of the product value proposition.

The authoring-time architecture is responsible for turning ordinary C# source code into:
- validated LM program definitions,
- deterministic generated metadata,
- and build-time artifacts that the runtime and compiler can consume safely.

** Core Principle
The framework has /three distinct compilation stages/:

1. /C# compilation understanding/
   - Roslyn parses and semantically understands authored source code.

2. /Authoring-time lowering/
   - analyzers validate the program shape,
   - source generators emit descriptors, manifests, and boilerplate adapters.

3. /LM program compilation/
   - the framework compiler explores variants, evaluates them, and emits a compiled artifact.

These stages must remain distinct.

** Why This Matters
This layer is the bridge between:
- "developer-authored code"
and
- "machine-operable program structure."

Without this layer, the framework would devolve into:
- reflection-heavy discovery,
- ad hoc runtime conventions,
- weak IDE feedback,
- and opaque program metadata.

That would undermine one of the core advantages of doing this in .NET.

** Build-Time Representation Pipeline
#+begin_example
C# source
  -> Roslyn semantic model
  -> analyzers
  -> source generators
  -> generated descriptors / manifests
  -> LM Program IR
#+end_example

This pipeline must be treated as a first-class architectural flow, not as an implementation detail.

** Roslyn Responsibilities
Roslyn is the /authoring compiler substrate/ for this framework.

Roslyn must provide:
- semantic understanding of signatures and programs,
- structural validation through analyzers,
- code fixes for high-value authoring mistakes,
- input to source generators.

Roslyn owns:
- source-level semantic validation,
- developer-facing diagnostics,
- code fix opportunities,
- access to the semantic model used by generators.

Roslyn must /not/ own:
- runtime execution,
- LM program compilation / optimization,
- artifact emission for compiled variants,
- business-task execution semantics.

Roslyn exists to understand authored source and guide developers before runtime.

** Analyzer Responsibilities
Analyzers are part of the product UX.
They are not secondary tooling.

Analyzers must:
- detect invalid or unsupported authoring shapes,
- report framework-specific diagnostics,
- prevent common architectural mistakes from reaching runtime,
- improve onboarding for junior developers.

Analyzers should target:
- missing field descriptions,
- missing or empty instructions,
- duplicate step names,
- non-deterministic step names,
- unsupported field types,
- unsupported graph shapes,
- invalid use of step APIs.

Analyzers must /not/:
- generate descriptors,
- execute model calls,
- run optimizations,
- substitute for runtime validation where live data is required.

** Code Fix Responsibilities
Code fixes are optional in breadth but important in principle.

The MVP should include a small number of high-value fixes such as:
- add missing description placeholder,
- normalize invalid attribute usage,
- suggest extracting a deterministic step name constant.

Code fixes must remain narrowly scoped and safe.

** Source Generator Responsibilities
Source generators are the /authoring-time lowering layer/.

They must:
- discover framework-authored types,
- lower authored declarations into deterministic generated outputs,
- eliminate excessive runtime reflection needs,
- generate metadata that becomes input to the LM Program IR and runtime/compiler layers.

Source generators own:
- signature descriptor generation,
- program descriptor generation,
- search-space descriptor generation,
- registration helper generation,
- optional manifest generation.

Source generators must /not/:
- run model calls,
- run evaluation,
- run optimization,
- invent runtime behavior,
- become the canonical home of program semantics.

The source generators are responsible for /materialization/, not /execution/ or /selection/.

** Generated Outputs Are Part of the Product
Generated outputs are first-class framework artifacts.
They are not incidental byproducts.

The framework must treat generated outputs as:
- deterministic,
- inspectable,
- testable,
- and part of the developer experience.

Expected generated outputs include:
- ~<SignatureName>.g.cs~
- ~<ProgramName>.g.cs~ (includes embedded metadata as C# constants/records)
/Note: A separate ~.dspyinfo.json~ manifest is NOT recommended. Source generators/
/cannot reliably flow non-.cs files into build outputs. Instead, generate metadata/
/as C# records/constants inside the ~.g.cs~ file, using ~file class~ to keep them/
/out of public API. This follows the precedent set by System.Text.Json source/
/generation (~JsonSerializerContext~) which embeds all metadata in C#./
- search-space metadata helpers
- registration helpers

These outputs should be visible in:
- tests,
- demos,
- and documentation.

** Signature Generation Contract
For each ~[LmpSignature]~ type, generation must emit:
- one deterministic signature descriptor,
- one deterministic registration or metadata entry,
- field metadata preserving names, descriptions, directions, and CLR type names,
- generator diagnostics if the type cannot be safely lowered.

The generated output must /not/ depend on runtime state.

** Program Generation Contract
For each ~[LmpProgram]~ type, generation must emit:
- one deterministic program descriptor,
- one deterministic search-space descriptor,
- one deterministic metadata manifest or equivalent inspection output,
- stable step identities derived from stable authored names.

Program generation must preserve the authored logical structure without inventing hidden runtime-only behavior.

** Build-Time vs Runtime Rule
Build-time owns:
- semantic understanding,
- descriptor generation,
- metadata generation,
- structural diagnostics.

Runtime owns:
- execution,
- binding,
- model invocation,
- trace capture,
- evaluator invocation.

Compiler owns:
- variant generation,
- trial orchestration,
- scoring,
- selection,
- artifact emission.

These boundaries must remain clean.

** MVP Role
In MVP, the authoring-time architecture must prove:
- authored signatures can be discovered,
- authored programs can be discovered,
- deterministic descriptors can be generated,
- high-value diagnostics can be surfaced.

** Future Role
After MVP, this layer may expand to include:
- richer code fixes,
- more expressive generated manifests,
- stronger AOT-oriented generation,
- better IDE integration,
- stronger static graph validation,
- richer authoring-time policy checks.

** What This Layer Is Not
This layer is not:
- the runtime,
- the optimizer,
- the canonical internal IR,
- or the deployment artifact system.

It is the /build-time lowering and guidance layer/.

#+begin_src csharp
using LMP;

[LmpProgram("support-triage")]
public partial class SupportTriageProgram : LmpProgram<TicketInput, TriageResult>
{
    public override ProgramGraph Build()
    {
        var retrieveKb = Step.Retrieve(
            name: "retrieve-kb",
            from: input => input.TicketText,
            topK: 5);

        var retrievePolicy = Step.Retrieve(
            name: "retrieve-policy",
            from: input => input.TicketText,
            topK: 3);

        var triage = Step.Predict<TriageTicket>(
            name: "triage",
            bind: (input, ctx) => new TriageTicket
            {
                TicketText = input.TicketText,
                AccountTier = input.AccountTier,
                KnowledgeSnippets = ctx.OutputOf(retrieveKb).Documents,
                PolicySnippets = ctx.OutputOf(retrievePolicy).Documents
            });

        var groundedness = Step.Evaluate(
            name: "groundedness-check",
            after: triage,
            evaluator: new GroundednessEvaluator());
            // Note: GroundednessEvaluator is a real class from
            // Microsoft.Extensions.AI.Evaluation.Quality.
            // BuiltInEvaluators factory does NOT exist — instantiate directly.

        var policy = Step.Evaluate(
            name: "policy-check",
            after: triage,
            evaluator: new CustomPolicyEvaluator("support-policy"));
            // Note: PolicyCompliance evaluator does NOT exist in M.E.AI.
            // Must be implemented as a custom evaluator for the framework.

        var repair = Step.If(
            name: "repair-if-needed",
            condition: ctx =>
                ctx.ScoreOf(groundedness) < 0.90 || !ctx.Passed(policy),
            then: Step.Repair<TriageTicket>(
                name: "repair-triage",
                usingFeedbackFrom: new[] { groundedness, policy }));

        return Graph
            .StartWith(retrieveKb)
            .Then(retrievePolicy)
            .Then(triage)
            .Then(groundedness)
            .Then(policy)
            .Then(repair)
            .Return(ctx => new TriageResult
            {
                Severity = ctx.Latest<TriageTicket>().Severity,
                RouteToTeam = ctx.Latest<TriageTicket>().RouteToTeam,
                DraftReply = ctx.Latest<TriageTicket>().DraftReply,
                Escalate = ctx.Latest<TriageTicket>().Escalate,
                GroundednessScore = ctx.ScoreOf(groundedness),
                PolicyPassed = ctx.Passed(policy)
            });
    }
}

** Canonical Compile Spec

using LMP.Compilation;

var compileSpec = CompileSpec
    .For<SupportTriageProgram>()
    .WithTrainingSet("data/support-triage-train.jsonl")
    .WithValidationSet("data/support-triage-val.jsonl")
    .Optimize(search =>
    {
        search.Instructions(step: "triage");
        search.FewShotExamples(step: "triage", min: 0, max: 6);
        search.RetrievalTopK(step: "retrieve-kb", min: 3, max: 10);
        search.RetrievalTopK(step: "retrieve-policy", min: 2, max: 6);
        search.Model(step: "triage", allowed: ["gpt-4.1-mini", "gpt-4.1"]);
        search.Temperature(step: "triage", min: 0.0, max: 0.7);
    })
    .ScoreWith(Metrics.Weighted(
        ("routing_accuracy", 0.35),
        ("severity_accuracy", 0.25),
        ("groundedness", 0.20),
        ("policy_pass_rate", 0.20)))
    .Constrain(rules =>
    {
        rules.Require("policy_pass_rate == 1.0");
        rules.Require("p95_latency_ms <= 2500");
        rules.Require("avg_cost_usd <= 0.03");
    })
    .UseOptimizer(Optimizers.RandomSearch());

* 9. Build-Time Responsibilities

** Build-Time Principle
Compile-time facilities must produce structure and diagnostics.
They must not attempt to perform runtime optimization.

** Three-Layer Build Architecture

LMP uses three build layers, mirroring how EF Core, Razor, Blazor, and Protobuf integrate with ~dotnet build~.

*** Layer 1: Source Generator (inside Roslyn — during ~dotnet build~)
Runs inside the C# compiler. Discovers ~[LmpSignature]~ and ~[LmpProgram]~ types,
validates structure, emits descriptors and binding code, reports diagnostics (LMP001–LMP007).
Provides real-time IDE feedback (red squiggles). Cannot access filesystem or run tools.

*** Layer 2: MSBuild Targets (after CoreCompile — during ~dotnet build~)
Runs after the C# compiler finishes, before build completes.
Three targets:
- ~LmpEmitIr~ (AfterCompile) — reads compiled assembly, extracts ProgramDescriptor instances,
  emits IR JSON to ~obj/lmp/~. This is the "program graph as data" step.
- ~LmpValidateGraph~ (AfterCompile, after LmpEmitIr) — validates IR completeness:
  all bindings resolved, no dangling step references, constraint predicates type-check.
- ~LmpEmbedArtifact~ (PrepareForPublish) — during ~dotnet publish~, copies compiled
  artifact to output directory or embeds as resource.

Implemented as ~Microsoft.Build.Utilities.Task~ subclasses, shipped via ~buildTransitive/~
folder in the NuGet package. Automatically imported when consuming project references LMP.

Precedent: EF Core
- Source: https://github.com/dotnet/efcore/blob/main/src/EFCore.Tasks/buildTransitive/Microsoft.EntityFrameworkCore.Tasks.targets
- ~CoreCompileDependsOn~ → ~_EFPrepareForCompile~ (pre-compile)
- ~TargetsTriggeredByCompilation~ → ~_EFGenerateFilesAfterBuild~ (post-compile)

*** Layer 3: CLI Tool (explicit ~dotnet lmp compile~ — costs $$$)
Runs only when developer explicitly invokes. Calls LM APIs (costs real dollars),
runs optimization trials, is non-deterministic, takes minutes to hours.
Must NEVER run during ~dotnet build~.

Precedent: EF Core separates the same way:
- ~dotnet build~ → compiles DbContext (implicit, free, deterministic)
- ~dotnet ef dbcontext optimize~ → generates compiled models (explicit, expensive)

** Why Layer 2 Matters
Without MSBuild targets, developers must run ~dotnet lmp compile~ (calling LM APIs,
costing money, taking minutes) just to discover a typo in a step name. With Layer 2,
~dotnet build~ catches ALL structural errors instantly — the CLI is only needed for
optimization.

** Analyzer Responsibilities
MVP analyzers must include:
- missing field description,
- missing or invalid signature instructions,
- unstable / non-constant step names,
- duplicate step names,
- unsupported cyclic graph authoring patterns,
- unsupported output field types.

** Source Generator Responsibilities
The source generator must:
- discover ~[LmpSignature]~ types,
- discover ~[LmpProgram]~ types,
- generate signature descriptors,
- generate program descriptors,
- generate search-space descriptors,
- generate minimal registration helpers,
- generate optional diagnostic-friendly manifests.

** MSBuild Target Responsibilities
The MSBuild targets must:
- emit IR JSON from compiled assembly metadata,
- validate program graph structural completeness,
- embed compiled artifacts during publish,
- report warnings/errors via MSBuild diagnostic infrastructure,
- ship via ~buildTransitive/~ in the NuGet package.

** Build-Time Outputs
Expected generated artifacts:
- ~<SignatureName>.g.cs~ (source generator)
- ~<ProgramName>.g.cs~ (source generator)
- ~obj/lmp/<ProgramName>.ir.json~ (MSBuild target)
- ~obj/lmp/validation-results.json~ (MSBuild target)
- metadata helper classes (source generator)

** What Must Not Happen at Build Time
Build time must not:
- run optimization trials,
- call language models,
- require online services,
- mutate datasets,
- emit runtime-specific prompt variants.

* 10. Runtime Responsibilities

** Runtime Principle
The runtime executes authored or compiled LM programs against live inputs.

** Runtime Inputs
The runtime consumes:
- authored program metadata,
- generated descriptors,
- runtime services,
- optional compiled artifact,
- program input.

** Step Execution Semantics
*** Retrieve Step
A retrieve step:
- takes a query from input or prior context,
- calls a retrieval provider abstraction,
- returns a document list.

MVP may implement retrieval with a stub or pluggable interface.

*** Predict Step
A predict step:
- binds data to a signature input object,
- assembles a prompt from descriptor + active variant,
- calls ~IChatClient~,
- parses structured output,
- stores output in context.

*** Evaluate Step
An evaluate step:
- consumes a prior output,
- runs an evaluator,
- stores scores and pass/fail metadata in context.

*** If Step
An if step:
- reads scores or context values,
- determines whether to execute the nested branch.

*** Repair Step
A repair step:
- reuses the same signature as a predict-like step,
- includes evaluation feedback in the prompt context,
- emits a replacement candidate output.

** Context Model
The runtime context must support:
- reading prior step outputs,
- reading latest output of a type,
- reading evaluator scores,
- reading pass/fail results,
- trace accumulation,
- cancellation propagation.

** Error Handling
MVP error rules:
- step exceptions fail the program unless explicitly handled,
- evaluation failures are recorded in trace,
- parsing failures surface as structured exceptions,
- failed repair does not silently replace original output.

** Trace Collection
Every step execution must produce a trace record.
Trace must capture enough information for:
- debugging,
- evaluation review,
- and compile report provenance.

* 10A. Optimization and Constraint Architecture

** Purpose
This section defines the architecture of program optimization and constraint enforcement.

This section exists to make the compile story precise and to remove ambiguity about:
- what is being optimized,
- how constraints apply,
- what the compiler owns,
- and what optimizer backends do.

** Core Principle
The optimizer backend is /not/ the compiler.

The compiler owns:
- search-space construction,
- objective definition,
- constraint enforcement,
- trial execution orchestration,
- selection of the best valid variant,
- compile report generation,
- artifact emission.

The optimizer backend only owns:
- candidate proposal.

This separation is mandatory.

** Optimization Pipeline
#+begin_example
LM Program IR
  -> Search Space Extraction
  -> Constraint Model
  -> Objective Model
  -> Candidate Proposal (optimizer backend)
  -> Trial Execution
  -> Evaluation
  -> Constraint Checking
  -> Selection
  -> Compiled Artifact
#+end_example

This is the canonical compile pipeline.

** What Is Being Optimized
The MVP optimizes /program variants/, not raw prompt strings in isolation.

A program variant is a complete assignment over a defined tunable subset of the LM program.

In MVP, tunables may include:
- instruction variant,
- few-shot example count,
- few-shot example selection,
- retrieval ~topK~,
- model choice,
- temperature.

The compiler must treat these as /program-level tunables/, not ad hoc runtime options.

** Optimization Layers
There are three conceptual optimization layers in the architecture.

*** 1. Structural Tuning
This changes explicit program parameters.

Examples:
- choose instruction variant,
- choose few-shot examples,
- choose retrieval top-k,
- choose model,
- choose temperature.

This is the core MVP optimization layer.

*** 2. Feasibility / Constraint Enforcement
This ensures that candidates satisfy required limits and rules.

Examples:
- policy pass rate must be 1.0,
- latency budget must not be exceeded,
- cost must not exceed threshold,
- incompatible choices must not be allowed together.

This is first-class in MVP.

*** 3. Search Guidance
This determines which candidate to try next.

Examples:
- random search,
- bounded grid search,
- hybrid search,
- later: AutoML, probabilistic, learned, symbolic-assisted strategies.

This layer is intentionally simple in MVP.

** Candidate / Trial / Compile / Artifact Semantics

*** Candidate
A candidate is a provisional assignment of tunable parameter values for one program.

A candidate is not yet selected or deployable.

*** Trial
A trial is the execution of one candidate on a dataset slice or evaluation set, producing:
- metrics,
- constraint outcomes,
- diagnostics,
- timing / cost data.

*** Compile
A compile is a bounded series of trials followed by:
- valid-candidate selection,
- report generation,
- artifact generation.

*** Compiled Artifact
A compiled artifact is the selected, versioned, deployable result of compilation.

** Constraint Model
Constraints are first-class framework concepts.
They must not be treated as ad hoc strings internally.

The public API may allow a compact DSL-like authoring experience, but internally constraints must lower into a typed model.

** Typed Constraint Shape
Conceptual model:

** Constraint Categories
*** Hard Constraints
Hard constraints must pass for a candidate to remain valid.

If a hard constraint fails:
- the candidate is invalid,
- it cannot be selected as best valid,
- and the report must record the violation.

Examples:
- ~policy_pass_rate == 1.0~
- ~p95_latency_ms <= 2500~
- ~avg_cost_usd <= 0.03~

*** Soft Constraints
Soft constraints are optional in MVP.
They may be recorded but need not influence selection in MVP unless explicitly supported.

Examples:
- prefer lower cost if primary score ties,
- prefer lower latency if quality remains comparable.

MVP may omit soft-constraint-driven ranking logic, but the architecture should preserve room for it.

** Constraint Timing
Constraints may act in multiple phases.

*** Static / Pre-Execution Constraints
These are checked before a trial runs.

Examples:
- parameter ranges,
- incompatible combinations,
- invalid candidate shapes.

MVP support:
- yes, but narrow in scope.

*** Runtime / Post-Execution Constraints
These are checked after metrics are collected.

Examples:
- latency,
- cost,
- policy pass rate,
- groundedness threshold.

MVP support:
- required.

** Framework Constraints vs Business Constraints
The architecture must distinguish:

*** Framework Constraints
These govern the validity of authored or compiled program structure.

Examples:
- stable step ids required,
- graph must be acyclic in MVP,
- tunable parameter must reference a valid step,
- unsupported output types are invalid.

These are not business optimization rules.
They are system-validity rules.

*** Business Constraints
These define the user’s optimization boundaries.

Examples:
- policy pass rate must equal 1.0,
- p95 latency <= 2500,
- avg cost <= 0.03.

These are compile-time decision rules over candidate behavior.

** Objective Model
The objective model defines what "better" means among /valid/ candidates.

MVP objective model:
- weighted metric aggregation.

Conceptual example:
#+begin_src csharp
public sealed record WeightedObjective(
    IReadOnlyList<(string MetricName, double Weight)> Metrics);
#+end_src

Example metrics:
- routing accuracy,
- severity accuracy,
- groundedness,
- policy pass rate.

** Selection Semantics
The MVP compiler must use the following decision order:

1. reject candidates that violate hard constraints,
2. rank remaining candidates by objective score,
3. break ties deterministically,
4. select the best valid candidate,
5. if no valid candidate exists, emit a failure report and no approved artifact.

This order is mandatory.

** Tie-Breaking
MVP tie-breaking should be deterministic and simple.
Recommended tie-break order:
1. lower cost,
2. lower latency,
3. lexicographically smaller variant id.

This prevents unstable compile results.

** Trial Budget Model
Optimization itself consumes time and money.
The compiler must therefore support a bounded optimization budget.

MVP required budget controls:
- maximum number of trials.

MVP recommended budget controls:
- maximum elapsed time.

Future budget controls may include:
- maximum optimization cost,
- dynamic early stopping.

** Optimizer Backend Role
The optimizer backend only proposes candidates.
It does not own:
- trial execution,
- evaluation,
- constraint enforcement,
- final selection,
- artifact generation.

Conceptual interface:
#+begin_src csharp

}
#+end_src

This interface is conceptual and may evolve, but the ownership boundary must remain.

** MVP Backend Strategy
The MVP backend should be intentionally simple.

Accepted MVP strategies:
- bounded random search,
- bounded grid search,
- simple hybrid search.

The purpose of the MVP backend is to prove:
- search space extraction,
- candidate application,
- trial execution,
- scoring,
- constraint enforcement,
- selection,
- artifact emission.

It is /not/ intended to prove state-of-the-art search.

** Why the MVP Backend Is Intentionally Boring
The architecture must first prove compiler legitimacy:
- the IR is real,
- the runtime is real,
- the evaluation loop is real,
- constraints are real,
- artifact emission is real.

A sophisticated optimizer built on a weak architecture would be misleading.
Therefore the MVP backend is intentionally boring so the architecture can be validated first.

** Z3 Position in the Architecture
Z3 is relevant for advanced scenarios, but is OVERKILL for the MVP.

MVP constraints are simple threshold checks:
- ~GroundednessScore >= 0.8~ (metric threshold)
- ~AvgLatency <= 2000ms~ (latency bound)
- ~CostPerCall <= 0.05~ (budget cap)

These are evaluable with lambda predicates — basic ~if~ statements.
Z3 is an SMT solver designed for SAT problems and theorem proving.
It should NOT be a core dependency for the MVP.

The recommended MVP approach:
- Constraints are ~Func<TrialMetrics, bool>~ predicates
- Evaluated at runtime after each trial
- Simple, debuggable, zero external dependency

The recommended post-MVP role for Z3 is:
- constraint co-processor for multi-constraint interaction analysis,
- static feasibility checker (is this constraint set satisfiable?),
- Pareto frontier computation (which constraints conflict?),
- optional candidate pruning engine.

The right early framing is:

#+begin_example
Compiler orchestration
  + simple optimizer backend
  + optional Z3-assisted feasibility pruning
#+end_example

This is powerful while keeping the architecture stable.

** Future Optimizer Backend Roadmap
The architecture must preserve room for richer backends.

*** Random / Hybrid Search Backend
Role:
- MVP backend.

*** Z3-Assisted Constraint Backend
Role:
- early post-MVP enhancement,
- static pruning,
- combinatorial feasibility support,
- candidate repair.

*** AutoML Backend
Role:
- structured mixed-parameter search.
/Note: ~Microsoft.ML.AutoML~ v0.23.0 uses SMAC-based Bayesian search over/
/numeric, categorical, and ordinal parameters. Genuine fit for hyperparameter/
/optimization over model choice, temperature, topK, and few-shot count./

*** Infer.NET-Guided Backend
Role:
- uncertainty-aware or probabilistic proposal guidance.
/Note: ~Microsoft.ML.Probabilistic~ (Infer.NET) is a Bayesian inference engine./
/It does NOT generate proposals or guide search directly. Realistic use:/
/posterior estimation over which parameter configurations are likely to satisfy/
/constraints, given observed trial results. This is Bayesian optimization,/
/not proposal generation. Framing should be corrected to "Bayesian surrogate/
/for trial outcome estimation" rather than "proposal guidance."/

*** TorchSharp Surrogate Backend
Role:
- learned ranking,
- learned proposal,
- surrogate-assisted search.
/Note: ~TorchSharp~ v0.106.1 provides PyTorch bindings for .NET. It does NOT/
/include built-in surrogate model implementations. To use TorchSharp here,/
/you would need to IMPLEMENT a surrogate model (e.g., a small MLP trained on/
/trial metrics to predict candidate quality). This is feasible but non-trivial./
/Framing should acknowledge that TorchSharp provides the SUBSTRATE, not/
/ready-made surrogates./

These future backends must remain pluggable behind the same compiler contract.

** What This Layer Must Not Become
This layer must not become:
- a solver-only project,
- a notebook experiment system,
- a backend-specific framework,
- or an optimizer whose contracts leak into the public authoring model.

The framework identity remains:
- compiler-oriented LM programming platform,
not
- search toolkit for prompts.

** MVP Guidance
The MVP must prove:
- candidates can be proposed,
- constraints can invalidate them,
- objective scoring can rank them,
- one best valid variant can be selected,
- and a compiled artifact can be emitted.

The MVP does /not/ need:
- advanced Bayesian search,
- learned surrogates,
- large-scale distributed search.

The demo should ideally show:
- at least one constraint-aware compile scenario,
- and optionally a narrow Z3-assisted pruning example,
so the framework does not appear to be merely brute-force prompt tuning.

* 23. Appendix D: Platform Gaps and .NET Team Recommendations

** Purpose
This section documents platform-level gaps discovered during validation.
Where .NET primitives are insufficient for the framework's needs, these gaps
represent potential feature requests or design-around strategies.

** Gap 1: Expression Tree Async Support
Expression trees (~Expression<Func<T>>~) cannot capture async lambdas or
statement bodies. This limits their use to single-expression data bindings.
Recommendation: if the C# team ever extends ~Expression<T>~ to support async
or statement bodies, the framework could use expression trees for richer
program structure inspection. Until then, use expression trees for data
binding ONLY and fluent builders for graph structure.

** Gap 2: Discriminated Union Types
Sealed abstract records + pattern matching work for step kind dispatch,
but are verbose (requires separate ~record~ declarations for each variant).
First-class ~union~ types in C# would make step algebra cleaner:
~type Step = Predict | Retrieve | Evaluate | If | Repair~
Recommendation: track the C# discriminated unions proposal.

** Gap 3: Source Generator Debugging
Debugging source generators is painful — no simple attach-and-step workflow.
Structured logging inside generators is ad hoc.
Recommendation: the framework should include a diagnostic dump mode that
writes generator intermediate state to disk for troubleshooting.

** Gap 4: Z3 NuGet Freshness
~Microsoft.Z3~ v4.12.2 lags Z3 v4.15.4 significantly.
Recommendation: pin to the available version and abstract behind an interface
so that future Z3 versions (or alternative solvers) can be swapped.

** Gap 5: IChatClient Observability
~IChatClient~ does NOT automatically emit OpenTelemetry traces.
The framework must wrap calls with explicit ~ActivitySource~ instrumentation.
Recommendation: file a feature request on ~dotnet/extensions~ for built-in
OTEL trace emission in ~ChatClientBuilder~.

** Gap 6: Interceptors — NOW STABLE in C# 14 / .NET 10
Interceptors graduated from experimental (C# 12) to fully stable in C# 14 / .NET 10.
They are now production-ready and used by ASP.NET Core Minimal APIs and OpenAPI.
LMP should use interceptors for Tier 3 binding analysis: intercept ~.Bind()~ calls,
read the lambda's syntax tree at compile time, and generate optimized binding code
+ serializable metadata. This eliminates expression tree limitations for compile-time
scenarios. Expression trees remain as a Tier 4 runtime-only fallback.
See Section 7.3 of the validation plan for the full three-tier binding architecture.
