* 0. Document Status

** Purpose
This document is the canonical implementation blueprint for LMP (Language Model Programs) —
a set of building blocks that bring LM program authoring, optimization, and runtime
execution into the .NET ecosystem, inspired by DSPy's core insight that LM programs
should be optimized programmatically rather than manually tuned.

LMP is building blocks, not a framework. Simple primitives compose naturally via
standard C# types. Source generators provide compile-time superpowers Python cannot
replicate.

** Provenance Note
LMP adopts DSPy's foundational concepts (Signatures, Predict, programmatic
optimization) and builds .NET-native equivalents using source generators, strong
typing, and the IChatClient abstraction from Microsoft.Extensions.AI.

Major .NET-native contributions include:
- separate TInput/TOutput design matching IChatClient's natural API shape,
- source-generated PromptBuilder, JsonTypeInfo, and GetPredictors,
- build-time diagnostics via Roslyn analyzers,
- AOT-safe serialization via System.Text.Json source generation,
- true parallelism in optimization (no GIL).

Target platform: .NET 10 LTS (C# 14), Aspire GA, Microsoft.Extensions.AI (IChatClient).

- Programs are Python classes inheriting ~dspy.Module~ with a ~forward()~ method.
- Optimizers include ~BootstrapFewShot~, ~BootstrapFewShotWithRandomSearch~,
  ~MIPROv2~, ~COPRO~, ~SIMBA~, ~KNNFewShot~, ~LabeledFewShot~.
- Persistence is via ~Module.save()~/~.load()~ — JSON serialization.
- There is no compile-time validation, no source generation, no typed prompt builders.

* 1A. The Business Case for Compiled LM Programs

** The Problem: Enterprise AI Is Bleeding Money

Every enterprise deploying LLM-powered applications faces four compounding cost drivers
that manual prompt engineering cannot solve:

*** Pain 1: Prompt Drift — Silent Failure at Enterprise Scale
When LLM behavior changes (model updates, prompt degradation, data distribution shift),
production applications silently fail. Unlike traditional software bugs, LLMs continue
to produce plausible outputs with degraded accuracy — failures go undetected until
significant business impact arises [1]. A ticket-triage system that starts misrouting
tickets triggers SLA violations, escalation labor, and customer churn.
Enterprises report 3–5× more debugging time per incident and 20–40% of AI budgets
wasted on unnecessary model swaps when root causes aren't systematically identified [2].
Manual prompt engineering provides no reproducibility guarantee — each "fix" is ad hoc,
untested against the original optimization criteria, and invisible to auditing.

*** Pain 2: Manual Optimization — Unquantified but Substantial Labor Cost
/LMP team estimate:/ A senior data scientist optimizing one LM program spends ~2 weeks
per optimization cycle across 10+ cycles annually: prompt variants, few-shot examples,
model selection, temperature tuning, evaluation runs. At blended rates ($150/hr ×
80 hrs × 10 cycles + infrastructure), this yields ~$154K/year /per program/.
This estimate is conservative — it excludes opportunity cost, review overhead, and
the labor of rebuilding optimization from scratch after model updates. The key insight
is not the specific dollar amount but that /this labor produces no reproducible artifact/
and must be repeated whenever the underlying model changes.

*** Pain 3: Governance Failure — EU AI Act Fines Up to 7% of Global Turnover
The EU AI Act Article 99 (effective August 2025) establishes tiered penalties [3]:
- Prohibited AI practices: up to €35M or 7% of global annual turnover (whichever is higher)
- High-risk AI obligations: up to €15M or 3% of global annual turnover
- Supplying incorrect information: up to €7.5M or 1% of global annual turnover
AI system providers must maintain technical documentation, traceability of
optimization decisions, and risk management systems with documented testing.
Manual prompt engineering produces NO audit trail, NO versioned configuration,
NO reproducible optimization history. Every manually tuned prompt is a compliance liability.

*** Pain 4: Enterprise AI Failure Rate — 70–85%
Enterprise AI investments show a consistent pattern of operational failure:
- 70–85% of GenAI deployment efforts fail to meet ROI expectations [4]
- Only ~25% of AI initiatives deliver expected ROI [5]
- 88–95% of AI pilots never reach production deployment [6][7]
- 42% of organizations abandoned most AI initiatives in 2025 (up from 17% in 2024) [8]
- GenAI spending projected to grow from $16B (2023) to $143B+ by 2027 (73% CAGR) [9]
- Most AI projects stall at "pilot purgatory" — technically functional but
  operationally immature, unable to demonstrate sustained business value [7]
The root cause is not technology inadequacy but /operational immaturity/: there is
no engineering discipline for managing LM programs as software — no build/test/deploy
lifecycle, no performance regression detection, no cost optimization automation [4].

* 2A. .NET Platform Primitive Map

** Purpose
This section maps specific .NET runtime, language, and SDK primitives to LMP
capabilities, organized by lifecycle phase. Every primitive listed here has been
validated as GA (production-ready) unless marked otherwise.

LMP's competitive advantage comes not from any single .NET feature, but from how
these primitives /converge/ across the full lifecycle. These convergences create
capabilities that Python fundamentally cannot replicate.

** Phase 1: Build Time — Author, Validate, Generate

*** IIncrementalGenerator (Roslyn)
Discovers ~[LmpSignature]~ output types at compile time.
Generates ~PromptBuilder<TIn, TOut>~, ~JsonTypeInfo<TOut>~, and ~GetPredictors()~ as ~.g.cs~ files.
Python equivalent: None — runtime metaclass magic only.

*** Roslyn Analyzers + CodeFixProvider
LMP001–LMP003 diagnostics surface errors in the IDE while typing.
Lightbulb fixes for missing descriptions.
Python equivalent: mypy/pylint (external, weaker, no semantic awareness).

*** file class (C# 11)
Generated PromptBuilder and JsonTypeInfo types stay out of public API surface.
Python equivalent: naming convention only.

*** static abstract members in interfaces (C# 11)
Enable source generators to implement factory methods on output types.
Python equivalent: static methods exist but interface enforcement does not.

*** Partial Properties (C# 13)
Source generators can implement properties on partial record types.
Python equivalent: None — no compile-time property generation.

*** Raw string literals (C# 11)
~"""..."""~ for multi-line prompt templates with zero escaping.
Python equivalent: triple-quoted f-strings (similar).

*** required keyword (C# 11) + Nullable Reference Types
Compile error if output field not set. ~string?~ vs ~string~ for optional/required.
Python equivalent: Runtime TypeError / ~Optional[str]~ (advisory only).

*** C# 14 Features (.NET 10 LTS)

**** Extension Members (C# 14)
Extension properties, static methods, and operators.
Enables richer fluent APIs without polluting core types.

**** Field-Backed Properties (C# 14)
~field~ keyword in property accessors for validation in setters.
Useful for output type validation.

**** Implicit Span Conversions (C# 14)
Zero-allocation prompt assembly APIs.

** Phase 2: Optimization — Evaluate, Search, Select

*** Records + with expressions
Safe immutable variant generation: ~v2 = v1 with { Temperature = 0.3 }~.
Compiler guarantees all fields are copied.
Python equivalent: ~dataclasses.replace()~ (no compiler guarantee).

*** FrozenDictionary / FrozenSet (.NET 8)
Immutable, cache-line-optimized lookup for predictor registries.
Python equivalent: ~types.MappingProxyType~ (not optimized).

*** INumber<T> / Generic Math (C# 11)
Generic metric aggregation.
Python equivalent: numpy (external dependency).

*** System.Threading.RateLimiting (.NET 7+)
~TokenBucketRateLimiter~ for API-safe optimization: prevents overwhelming LM APIs
during multi-trial optimization loops.
Python equivalent: ~tenacity~ (external).

*** Parallel.ForEachAsync (.NET 6+)
Bounded-concurrency parallel trial execution with ~MaxDegreeOfParallelism~.
Python equivalent: ~asyncio.Semaphore~ (GIL-limited for CPU work).

*** HybridCache (.NET 9, enhanced .NET 10)
Combined L1 (in-memory) + L2 (distributed) cache for LM response deduplication.
Same prompt → cached response across optimization trials.
Python equivalent: ~functools.lru_cache~ + Redis (manual wiring).

*** BackgroundService
Long-running optimization jobs with graceful cancellation via ~CancellationToken~.
Python equivalent: manual signal handling.

*** System.Numerics.Tensors.TensorPrimitives (.NET 9+)
SIMD-accelerated math operations — zero external dependency:
- ~TensorPrimitives.CosineSimilarity(span1, span2)~ — embedding similarity for retrieval
- ~TensorPrimitives.Sum()~, ~Average()~, ~Max()~ — metric aggregation in optimizers
Python equivalent: numpy (external dependency).

** Phase 3: Runtime — Execute, Trace, Observe

*** IChatClient Middleware Pipeline (Microsoft.Extensions.AI)
M.E.AI provides built-in middleware via ~ChatClientBuilder~:
- ~UseOpenTelemetry()~ — automatic OTEL spans for every LM call
- ~UseDistributedCache()~ — response caching via ~IDistributedCache~
- ~UseLogging()~ — structured prompt/completion logging
- ~UseFunctionInvocation()~ — automatic tool calling dispatch
Python equivalent: None — no middleware pipeline for LLM calls.

*** Keyed DI Services (.NET 8)
~[FromKeyedServices("triage")] IChatClient client~ — multiple named model configs.
Python equivalent: manual registry dictionary.

*** True Parallelism (no GIL)
~Task.WhenAll~ runs multiple LM calls concurrently on separate threads.
BestOfN and BootstrapRandomSearch benefit from true parallelism.
Python equivalent: asyncio is limited by the GIL for CPU-bound scoring.

*** IAsyncEnumerable<T>
Token streaming as first-class iteration.
Python equivalent: library-specific streaming.

*** ActivitySource + Activity (System.Diagnostics)
Per-predictor distributed tracing with W3C Trace Context.
Tags for model, tokens, cost.
Python equivalent: manual dicts in DSPy.

*** Meter + Histogram / Counter
Custom metrics: ~lm.predict.latency_ms~, ~lm.cost.total_usd~, ~lm.tokens.input~.
Export to Prometheus, Grafana, Azure Monitor.
Python equivalent: manual instrumentation.

*** DataAnnotations + IValidatableObject
~[Required]~, ~[Range]~ for declarative LM output validation.
Output types may implement ~IValidatableObject~ for structured self-validation.
Python equivalent: Pydantic validators (similar).

*** [GeneratedRegex] (.NET 7+)
AOT-safe, pre-compiled regular expressions via source generation.
Python equivalent: ~re.compile()~ (interpreted).

*** Microsoft.Extensions.Resilience (Polly-based)
Composable resilience pipelines for LM API calls:
- Retry with exponential backoff + jitter
- Circuit breaker
- Timeout
Python equivalent: ~tenacity~ (retry only, no circuit breaker).

** Phase 4: Deployment — Save, Load

*** System.Text.Json + JsonSerializerContext
AOT-safe serialization with source-generated serializers. Zero reflection.

Python equivalent: ~json.dumps~ (no AOT story).

*** ConfigurationBinder Source Generator (.NET 8+)
AOT-safe configuration binding for model settings, temperature, etc.
Python equivalent: None (env var parsing is always dynamic).

*** Native AOT
Compiled programs as native executables: ~50ms cold start vs 2–5s for Python.
Source generators eliminate reflection for AOT compatibility.
Python equivalent: No equivalent.

** Phase 5: Tooling — Orchestrate, Observe

*** .NET Aspire (GA)
Dev-time orchestration: LM provider + metrics dashboard.
One ~dotnet run~, automatic OTEL dashboard.
Status: GA. Modern Lifecycle Policy.

*** Microsoft.Extensions.Hosting
CLI and services use identical DI/logging/config infrastructure.
Python equivalent: manual setup.

* 5. User-Facing Mental Model

** Output Signature
A /Signature/ is an ~[LmpSignature]~ attribute on a ~partial record~ output type.
/Provenance: Adopted from DSPy (~dspy.Signature~). DSPy uses a single class with/
/InputField and OutputField decorators. LMP separates input and output into distinct/
/C# types — matching how ~IChatClient.GetResponseAsync<T>()~ naturally works./

A signature contains:
- instructions (the task description),
- typed output fields with descriptions,
- the partial record is the output type.

The source generator reads this at build time and emits:
- ~PromptBuilder<TIn, TOut>~ — assembles ~ChatMessage[]~ from instructions + demos + input,
- ~JsonTypeInfo<TOut>~ — zero-reflection JSON serialization,
- diagnostics — IDE red squiggles for missing descriptions, non-serializable types.

** Input Type
An input type is a plain C# record. No LMP attributes needed — it's just data.
Descriptions come from ~[Description]~ on constructor parameters or XML doc comments.

** Predictor
~Predictor<TIn, TOut>~ is the core primitive. It binds an input type to an output type.
Contains learnable state: ~Demos~ (few-shot examples), ~Instructions~ (overridable),
~Config~ (temperature, etc.).
/Provenance: Adopted from DSPy (~dspy.Predict~). Extended with source-generated/
/PromptBuilder and typed Demos./

** Module
~LmpModule~ is a composable base class. Users override ~ForwardAsync()~ to compose
multiple predictors into an LM program.
/Provenance: Adopted from DSPy (~dspy.Module~ with ~forward()~). LMP mirrors this/
/exactly — no graphs, no IR, just a method that calls predictors./

Source generator emits ~GetPredictors()~ for zero-reflection predictor discovery.
~SaveAsync()~ / ~LoadAsync()~ use source-generated ~JsonSerializerContext~.

** Evaluator
An /Evaluator/ scores model outputs on a dataset using a metric function.
/Provenance: Adopted from DSPy (~dspy.Evaluate~). LMP integrates with/
/~Microsoft.Extensions.AI.Evaluation~ for production-grade evaluators/
/(groundedness, coherence, relevance) not available in DSPy./

** Optimizer
An /Optimizer/ runs a module on training data, collects successful traces,
and fills predictor parameters (demos, instructions) automatically.
/Provenance: Adopted from DSPy (~dspy.BootstrapFewShot~, etc.)./
/LMP optimizers return the same module type with parameters filled in./

** Compiled Parameters
Optimized parameters are saved as JSON via ~SaveAsync()~ and loaded via ~LoadAsync()~.
This is the "compiled artifact" — a versioned JSON file containing demos,
instructions, and config for each predictor in the module.
/Provenance: Extended from DSPy's ~Module.save()~/~.load()~. LMP uses source-generated/
/JsonSerializerContext for AOT-safe serialization./

* 5A. Authoring-Time Architecture: Source Generators and Analyzers

** Purpose
This section defines the build-time authoring architecture.

This architecture is one of the main reasons LMP belongs in .NET.
It is not optional polish — it is a core part of the product value proposition.

** Core Principle
LMP has /two distinct compilation stages/:

1. /C# compilation with source generation/
   - Roslyn parses output types with ~[LmpSignature]~,
   - source generator emits PromptBuilder, JsonTypeInfo, GetPredictors,
   - analyzers validate and report diagnostics.

2. /LM program optimization/
   - optimizers (BootstrapFewShot, etc.) run modules on training data,
   - successful traces fill predictor Demos,
   - optimized parameters are serialized via SaveAsync.

These stages must remain distinct. Stage 1 happens at ~dotnet build~ (free, deterministic).
Stage 2 happens when the developer explicitly runs optimization (costs money, non-deterministic).

** Source Generator Responsibilities

The source generator discovers ~[LmpSignature]~ output types and emits:

*** PromptBuilder<TIn, TOut>
Assembles ~ChatMessage[]~ from instructions, demos, and input fields.
Field names, types, and descriptions are baked in as string constants.
No runtime reflection.

*** JsonTypeInfo<TOut>
Zero-reflection JSON serialization via ~System.Text.Json~ source generation.
AOT-safe. Used by ~GetResponseAsync<TOut>()~ for structured output.

*** GetPredictors() on LmpModule
Discovers all ~Predictor~ fields in a module without walking dictionaries at runtime.
This is what DSPy's ~named_predictors()~ does via ~__dict__~ introspection in Python.

*** ChainOfThought extended output record
When ~ChainOfThought<TIn, TOut>~ is used, the source generator creates an extended
output record that adds a ~Reasoning~ field before the final answer fields.
The LM produces step-by-step reasoning, then the structured output.

** Analyzer Responsibilities
Analyzers are part of the product UX. They are not secondary tooling.

MVP analyzers (2–3 rules):
- LMP001: Missing ~[Description]~ on output properties → warning
- LMP002: Non-serializable output type → error
- LMP003 (future): Unreachable predictor in module → warning

** Input Field Description Sources
The source generator reads input descriptions from three sources, in priority order:
1. XML doc comments (~/// <param name="X">...</param>~)
2. ~[Description]~ on constructor parameters
3. ~[Description]~ on properties

** Generated Outputs Are Part of the Product
Generated outputs are first-class artifacts. They must be:
- deterministic,
- inspectable,
- testable,
- and part of the developer experience.

Expected generated outputs:
- ~PromptBuilder_TIn_TOut.g.cs~
- ~TOut_JsonTypeInfo.g.cs~ (or via JsonSerializerContext)
- ~TModule_GetPredictors.g.cs~

** Build-Time vs Runtime Rule
Build-time owns:
- PromptBuilder generation,
- JsonTypeInfo generation,
- GetPredictors generation,
- diagnostics (LMP001–LMP003).

Runtime owns:
- execution (ForwardAsync),
- model invocation via IChatClient,
- trace capture,
- assertion enforcement.

Optimization owns:
- running modules on training data,
- collecting successful traces,
- filling predictor Demos,
- selecting best parameters.

These boundaries must remain clean.

** Layer 0: .NET Platform (Already Exists — Zero LMP Code)

Everything below is provided by the .NET ecosystem. LMP builds on top, never re-implements.

| Capability           | .NET Component                    | What It Does for LMP                           |
|----------------------+-----------------------------------+------------------------------------------------|
| LM abstraction       | ~IChatClient~                     | Uniform interface to any LM provider           |
| Structured output    | ~GetResponseAsync<T>()~           | Deserializes LM output directly into C# types  |
| Middleware pipeline  | ~ChatClientBuilder~               | Caching, telemetry, logging, rate limiting      |

| Tool use             | ~AIFunction~ / ~FunctionInvokingChatClient~ | Function calling for ReAct agent tools |
| Configuration        | ~IOptions<T>~                     | Standard options pattern for model settings     |

Key insight: Layer 0 already solves LM calling, structured output, tool use, evaluation,
and middleware. LMP's job is to add /programmable optimization/ on top.

** Layer 1: Core Building Blocks (~LMP.Abstractions~ + ~LMP.Core~)

*** [LmpSignature("instructions")]
An attribute placed on a ~partial record~ output type. Entry point for the source generator.

*** Predictor<TInput, TOutput>
The core primitive. Binds an input type to an output type. Contains learnable state.
Internally: source-generated PromptBuilder → ChatMessage[] → IChatClient.GetResponseAsync<TOutput>().

*** LmpModule
Base class for composable LM programs. Users override ForwardAsync().
Source generator emits GetPredictors(). SaveAsync/LoadAsync use source-generated JsonSerializerContext.

*** Supporting Types
| Type                         | Purpose                                                       |
|------------------------------+---------------------------------------------------------------|
| ~Example<TInput, TLabel>~    | Training data record. ~WithInputs()~ extracts input portion.  |
| ~Trace~                      | Records (predictor, inputs, outputs) tuples for optimization. |
| ~LmpAssert~                  | Runtime assertion with retry: ~LmpAssert.That(result, pred)~  |
| ~LmpSuggest~                 | Soft assertion — logs warning, no retry.                      |
| ~IRetriever~                 | RAG interface: ~RetrieveAsync(query, k) → string[]~.          |

** Layer 2: Reasoning Strategies (~LMP.Modules~)

Thin wrappers around ~Predictor<TIn, TOut>~. Each is under 100 lines of code.

*** ChainOfThought<TIn, TOut>
Extends TOut with a Reasoning field. Source generator creates an extended output record
so the LM produces step-by-step reasoning before the final answer.
/Inspired by: dspy/predict/chain_of_thought.py (49 LOC in DSPy)./

*** BestOfN<TIn, TOut>
Runs N parallel predictions via ~Task.WhenAll~, returns the best by a reward function.
True parallelism — no GIL.

*** Refine<TIn, TOut>
Sequential improvement: predict → LM-generated critique → predict again with critique context.

*** ReActAgent<TIn, TOut>
Think → Act → Observe loop using M.E.AI's ~AIFunction~ for tools.
No custom tool abstraction needed — ~FunctionInvokingChatClient~ handles tool dispatch.
/Inspired by: dspy/predict/react.py (345 LOC in DSPy)./

*** ProgramOfThought<TIn, TOut> (post-MVP)
LM generates C# code → Roslyn scripting executes it → returns structured result.

** Layer 3: Optimization (~LMP.Optimizers~)

This is DSPy's core insight brought to .NET: LM programs should be optimized
programmatically, not manually tuned.

*** Evaluator
Runs a module on a dev set, scores with a metric function, aggregates results.
Uses ~Parallel.ForEachAsync~ for concurrent evaluation.

*** BootstrapFewShot
The core "compile" step. Runs a teacher module on the training set, collects
successful traces (where the metric passes), and fills ~predictor.Demos~ with
those traces as few-shot examples.
/Inspired by: dspy/teleprompt/bootstrap.py (~250 LOC in DSPy)./

*** BootstrapRandomSearch
BootstrapFewShot × N candidates with ~Task.WhenAll~. Returns the best.

*** MIPROv2 (post-MVP)
Bayesian optimization over both instructions and demos. DSPy's MIPROv2 uses
Optuna's TPE sampler. LMP could use ML.NET AutoML tuners as a backend.

*** IOptimizer
All optimizers implement this interface. All return the same module type with
parameters filled in — no new types created.

#+begin_src csharp
public interface IOptimizer
{
    Task<TModule> OptimizeAsync<TModule>(
        TModule module,
        IReadOnlyList<Example> trainSet,
        Func<object, object, float>? metric = null)
        where TModule : LmpModule;
}
#+end_src

** Separate TInput / TOutput Design

This is a deliberate departure from DSPy, where a single ~dspy.Signature~ class mixes
input and output fields. In LMP, input and output are separate C# types.

Why: M.E.AI's actual API is ~chatClient.GetResponseAsync<T>()~ where T is the output
type and input is ~ChatMessage[]~. They are naturally separate. Forcing them into one
type fights the platform.

* 9. Five Things Python Cannot Do

These are structural advantages of the .NET platform that no amount of Python library
code can replicate:

1. /Compile-time signature validation/ — Missing descriptions, invalid output types, and
   type mismatches surface as IDE red squiggles and build errors. In DSPy, these are
   runtime crashes.

2. /Zero-reflection predictor discovery/ — Source gen emits ~GetPredictors()~ at build time.
   DSPy's ~named_predictors()~ walks ~__dict__~ at runtime.

3. /Source-generated prompt builders/ — ~PromptBuilder<TIn, TOut>~ is a concrete class
   with field names baked in as constants. No string formatting at runtime.

4. /AOT-deployable LM programs/ — With source-gen JSON and no reflection, LMP modules
   can be published as native AOT binaries. ~50ms cold start vs. 2–5s for Python.

5. /True parallelism in optimization/ — ~Task.WhenAll~ + ~Parallel.ForEachAsync~ use real
   OS threads. Python's GIL serializes CPU-bound work, making BestOfN and
   BootstrapRandomSearch fundamentally slower.

* 10. Architectural Responsibility Map

** Purpose
This section defines /what goes where/ in the system.

** Architectural Center of Gravity
The central architectural facts are:

1. ~Predictor<TIn, TOut>~ is the core primitive.
2. ~LmpModule~ is the composition unit.
3. Source generators provide the build-time differentiator.
4. ~IChatClient~ from ~Microsoft.Extensions.AI~ is the runtime substrate.
5. ~M.E.AI.Evaluation~ is the scoring substrate.

Microsoft Semantic Kernel is now in MAINTENANCE MODE (no new features, critical bug
and security fixes only). Its successor is the Microsoft Agent Framework (MAF = SK +

competitive with LMP — they orchestrate agents; LMP optimizes LM programs.
The architectural layering is:
- Layer 1: Microsoft.Extensions.AI (IChatClient) — LLM abstraction
- Layer 2: LMP — LM program optimization
- Layer 3: MAF — Agent orchestration (optional consumer)
LMP depends ONLY on ~IChatClient~ from ~Microsoft.Extensions.AI~.

* 11. Optimization Architecture

** Purpose
This section defines how program optimization works.

** Core Principle
Optimization is a library feature, not a CLI-first workflow.
Developers call ~optimizer.OptimizeAsync(module, trainSet)~ in code.
A CLI wrapper is optional post-MVP tooling (Layer 4).

** What Is Being Optimized
The MVP optimizes /predictor parameters/, not raw prompt strings in isolation.

Optimizable parameters per predictor:
- few-shot demos (which examples to include),
- instructions (override the [LmpSignature] default),
- config (temperature, etc.).

** Optimization Pipeline
#+begin_example
Training data (Example<TInput, TLabel>[])
  → Run teacher module on each example
  → Collect successful traces (where metric passes)
  → Fill predictor.Demos with traced examples
  → Evaluate on dev set
  → Select best configuration
  → Return optimized module
#+end_example

** BootstrapFewShot
The core optimizer. Algorithm:
1. For each training example, run the module's ForwardAsync.
2. Score the output with the metric function.
3. If the metric passes, record the (input, output) pair as a demo trace.
4. Fill each predictor's Demos with the top-scoring traces (up to maxDemos).
5. Return the module with Demos filled.

** BootstrapRandomSearch
BootstrapFewShot × N candidates:
1. Run BootstrapFewShot N times with different random seeds / shuffles.
2. Evaluate each candidate on the dev set.
3. Return the candidate with the highest score.
Uses ~Task.WhenAll~ for true parallelism.

** MIPROv2 (post-MVP)
Bayesian optimization over both instructions and demos.
DSPy's MIPROv2 uses Optuna's TPE sampler (~35K LOC).
LMP could use ML.NET AutoML tuners as a backend.

** Budget Controls
MVP required:
- maximum number of trials (for BootstrapRandomSearch).
- maximum demos per predictor (for BootstrapFewShot).

** Selection Semantics
For BootstrapRandomSearch:
1. Evaluate all N candidates on dev set.
2. Rank by metric score.
3. Break ties deterministically (lower index).
4. Return the best candidate.

* 12. Save / Load (Artifact Model)

** Purpose
Optimized parameters must be persistable and loadable for production deployment.

** Design
~SaveAsync(path)~ serializes all predictor parameters (demos, instructions, config)
to a JSON file using source-generated ~JsonSerializerContext~. AOT-safe, zero reflection.

~LoadAsync(path)~ deserializes and applies parameters to the module's predictors.

** Compatibility
A saved file is valid for the same module type. The JSON includes the module type name
and version for validation.

** What Is Saved
Per predictor:
- instructions (string),
- demos (list of input/output pairs),
- config (temperature, etc.).

Module-level metadata:
- optimizer used,
- best score,
- timestamp,
- module type name,
- version.

** What Is Not Saved
- The module code itself (it's compiled C#),
- the IChatClient configuration (that's DI / infrastructure),
- training data (that's external).

* 18. Acceptance Criteria

* 22. What's Intentionally Excluded

The following concepts appeared in earlier drafts but have been dropped. They added
complexity without matching how DSPy actually works or how .NET developers naturally
build software:

| Dropped Concept                 | Why                                                                          |
|---------------------------------+------------------------------------------------------------------------------|
| Directed program graphs         | DSPy has no graph IR. Modules are Python classes with forward().             |
|                                 | LMP mirrors this with LmpModule.ForwardAsync().                             |
| Intermediate Representation     | DSPy has no IR layer. Descriptor types added indirection without value.      |
| MSBuild targets for validation  | No graphs → no graph validation. Source gen diagnostics are sufficient.     |
| Three-tier binding system       | Over-engineered. Predictor<TIn,TOut> with PromptBuilder covers all binding. |
| Hot-swap AssemblyLoadContext     | Premature. SaveAsync/LoadAsync with JSON is the right starting point.       |
| NuGet artifact packaging        | Not core architecture. Can be layered later if needed.                      |
| CLI as core architecture        | Optimization is a library feature, not a CLI-first workflow.                |
| 7+ diagnostics with code fixes  | Start with 2–3 essential diagnostics. Add more based on real usage.         |
| Three-layer build architecture  | One source generator, one build step. ~dotnet build~ does everything.       |
| Constraint model with Z3        | MVP constraints are lambda predicates. Z3 is post-MVP.                      |
| CompileSpec DSL                 | Optimization is API calls, not a separate spec language.                    |
| Step / Edge / Graph descriptors | No graph → no graph descriptors. Predictors are discovered via GetPredictors.|

* 24. Appendix B: Generated Code Example

#+begin_src csharp
// <auto-generated />
// Source generator output for Predictor<TicketInput, ClassifyTicket>

using System.ComponentModel;
using Microsoft.Extensions.AI;

namespace LMP.Generated;

file static class PromptBuilder_TicketInput_ClassifyTicket
{
    private const string Instructions =
        "Classify a support ticket by category and urgency";

    private static readonly (string Name, string Description)[] InputFields =
    [
        ("TicketText", "The raw ticket text"),
        ("AccountTier", "Customer plan tier"),
    ];

    private static readonly (string Name, string Description)[] OutputFields =
    [
        ("Category", "Category: billing, technical, account"),
        ("Urgency", "Urgency from 1-5"),
    ];

    public static ChatMessage[] Build(
        TicketInput input,
        IReadOnlyList<Demo<TicketInput, ClassifyTicket>>? demos = null)
    {
        var messages = new List<ChatMessage>();

        // System message with instructions and field descriptions
        messages.Add(new ChatMessage(ChatRole.System, BuildSystemPrompt()));

        // Demo pairs (few-shot examples)
        if (demos is not null)
        {
            foreach (var demo in demos)
            {
                messages.Add(new ChatMessage(ChatRole.User, FormatInput(demo.Input)));
                messages.Add(new ChatMessage(ChatRole.Assistant,
                    JsonSerializer.Serialize(demo.Output)));
            }
        }

        // Current input
        messages.Add(new ChatMessage(ChatRole.User, FormatInput(input)));

        return messages.ToArray();
    }

    private static string BuildSystemPrompt()
    {
        var sb = new StringBuilder();
        sb.AppendLine(Instructions);
        sb.AppendLine();
        sb.AppendLine("Input fields:");
        foreach (var (name, desc) in InputFields)
            sb.AppendLine($"  [[ ## {name} ## ]] {desc}");
        sb.AppendLine();
        sb.AppendLine("Output fields (respond as JSON):");
        foreach (var (name, desc) in OutputFields)
            sb.AppendLine($"  [[ ## {name} ## ]] {desc}");
        return sb.ToString();
    }

    private static string FormatInput(TicketInput input) =>
        $"[[ ## TicketText ## ]] {input.TicketText}\n[[ ## AccountTier ## ]] {input.AccountTier}";
}
#+end_src

* 25. Appendix C: MVP Non-Negotiables

The following are non-negotiable:
- use ~[LmpSignature]~ on output types (separate TInput / TOutput),
- use source generation for PromptBuilder, JsonTypeInfo, and GetPredictors,
- use ~Predictor<TIn, TOut>~ as the core primitive,
- use ~LmpModule~ with ~ForwardAsync()~ for composition,
- use analyzers for 2–3 core diagnostics,
- use ~IChatClient~ as the LM runtime substrate,
- integrate evaluation via Evaluator + metric functions,
- implement BootstrapFewShot and BootstrapRandomSearch,
- implement SaveAsync / LoadAsync for optimized parameters,
- keep the MVP story centered on ticket triage.

* 26. Appendix D: Platform Gaps

** Gap 1: Source Generator Debugging
Debugging source generators is painful — no simple attach-and-step workflow.
Recommendation: include a diagnostic dump mode that writes generator state to disk.

** Gap 2: Discriminated Union Types
Sealed abstract records + pattern matching work but are verbose.
First-class ~union~ types in C# would make step kinds cleaner.
Recommendation: track the C# discriminated unions proposal.

* 27. Final Instruction to Implementers

Do not begin by building an advanced optimizer.
Do not begin by building a CLI.
Do not begin by building an IDE extension.
Do not begin by building graphs, IRs, or MSBuild targets.

Begin by making the canonical ticket triage story real with:
- separate TInput / TOutput types,
- ~Predictor<TIn, TOut>~ calling IChatClient,
- source-generated PromptBuilder,
- ~LmpModule~ with ForwardAsync composing two predictors,
- ~Evaluator~ scoring on a dataset,
- ~BootstrapFewShot~ filling Demos from training data,
- ~SaveAsync~ / ~LoadAsync~ for optimized parameters.

That is the MVP.
